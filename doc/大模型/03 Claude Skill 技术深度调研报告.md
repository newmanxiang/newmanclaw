# Claude Skill 技术深度调研报告

## 调研概述

本报告从第一性原理出发，深度剖析 Anthropic Claude Skill 技术体系，对比其与 MCP 的核心差异，拆解其本质定义，厘清其与大模型、工具的底层关系，并详细分析开源实现 OpenClaw 中的工程落地细节，为 AI Agent 能力扩展的技术选型与开发实践提供完整的技术参考。

---

## 一、为什么是 Skill 而非 MCP？—— 定位与效率的底层差异

在 Anthropic 的 AI 生态中，MCP（Model Context Protocol）与 Skill 是两个定位完全不同的能力扩展机制，二者并非替代关系，而是解决不同层级问题的 \&\#34;左右手\&\#34;，但在绝大多数通用场景下，Skill 展现出了远高于 MCP 的适配性与效率优势。

### 1\.1 核心定位的本质区别

|维度|Skill|MCP|
|---|---|---|
|**本质定义**|标准化工作流指令包、领域知识手册|跨服务通信协议、外部集成标准|
|**架构层级**|提示 / 知识层（大脑层）|集成 / 连接层（手脚层）|
|**核心目标**|教 AI\&\#34;如何做\&\#34;（方法论）|让 AI\&\#34;能做什么\&\#34;（能力接入）|
|**依赖组件**|仅需 Markdown 文件，无需额外服务|需要独立运行的 MCP Server，维护双向通信|

Anthropic 官方明确指出：**Skills are at the prompt/knowledge layer, whereas MCP is at the integration layer\.**

- MCP 解决的是 \&\#34;连接\&\#34; 问题：让 AI 突破自身数据局限，访问外部世界的工具、数据源与服务，比如对接 GitHub、Slack、数据库这类需要实时交互的外部系统。

- Skill 解决的是 \&\#34;方法论\&\#34; 问题：为 AI 注入专业的流程知识，教 AI 如何标准化完成特定任务，比如按照公司规范写代码审查报告、按照固定流程处理 PDF 文件。

### 1\.2 工程效率的显著优势

#### Token 消耗的极致优化

MCP 的最大痛点是上下文爆炸：单个 GitHub MCP 的工具定义就会消耗上万 Token，加载几个 MCP 服务就会直接占满模型的上下文窗口，导致成本飙升、模型指令遵循能力下降。

而 Skill 通过**渐进式披露**机制实现了极致的 Token 效率：

- 100 个 Skill 的元数据总消耗仅～5000 Token，日常状态下仅占用极小的上下文空间

- 仅当模型判断需要使用某个 Skill 时，才会动态加载该 Skill 的详细指令与资源

- 实测数据显示，相同能力下，Skill 的 Token 消耗仅为 MCP 的 1/10 甚至更低

#### 开发与迁移成本的碾压性优势

- **开发成本**：创建一个 Skill 仅需编写一个 Markdown 文件，5 分钟即可完成；而开发 MCP Server 需要理解协议、实现端点、处理通信，半天起步。

- **跨模型可移植性**：Skill 本质是 Markdown 指令，只要模型具备基础的上下文学习能力，就能直接使用，无论是 Claude、GPT 还是 Gemini、开源大模型都能无缝兼容；而 MCP 是 Anthropic 定义的专属协议，跨模型基本无法复用。

### 1\.3 选型边界：什么时候用什么？

- **优先选择 Skill 的场景**：标准化工作流、本地文件处理、快速验证想法、跨模型通用的能力封装

- **必须使用 MCP 的场景**：实时 API 交互、复杂认证流程、双向通信的企业级集成

最佳实践是二者协同：MCP 负责提供底层的原子能力接入，Skill 负责基于这些能力封装标准化的工作流，MCP 负责 \&\#34;能不能\&\#34;，Skill 负责 \&\#34;该不该\&\#34;。

## 二、Skill 的第一性原理本质 —— 动态提示词注入的元工具架构

从第一性原理出发，Skill 的本质可以概括为：**基于提示词的动态上下文注入与元工具架构（Prompt\-based Dynamic Context Injection \&amp; Meta\-Tool Architecture）**。

### 2\.1 破除认知误区：Skill 不是什么？

Anthropic 官方反复强调，Skill 既不是插件，也不是函数调用，更不是可执行代码：

- 它不是一段在模型之外运行的程序：Skill 本身不会直接执行任何操作

- 它不是传统的 Function Calling：它不会像工具一样返回执行结果

- 它不是静态的 System Prompt：它不是一次性加载的全局设定

### 2\.2 Skill 的真正本质

Skill 是一份**可动态插拔的领域专家知识包**，它的核心逻辑是：

> 当 Skill 被调用时，系统并不会去 \&\#34;运行\&\#34; 这个 Skill，而是读取这个文件，将其中的大量指令、工作流和知识 \&\#34;展开\&\#34; 并 \&\#34;植入\&\#34; 到模型的对话上下文中，瞬间把通用的 Claude 变成解决该问题的领域专家，让 Claude 自己去解决问题。
> 
> 

它的核心架构是**元工具（Meta\-Tool）**：

- 日常状态下，模型的工具列表中只有一个名为`Skill`的元工具，这个元工具的描述里，只包含所有可用 Skill 的**名称和 100 字左右的简短描述**

- 模型完全依靠自身的语言理解能力，将用户的意图与这些 Skill 的描述进行匹配，决策是否需要调用某个 Skill

- 这里没有任何硬编码的路由、分类器或者正则匹配，所有的决策完全发生在 Transformer 的前向传播中，是纯 LLM 的推理过程

### 2\.3 核心机制：双重上下文注入

当一个 Skill 被选中时，它会执行两个层面的上下文修改，这是 Skill 最核心的创新：

#### A\. 对话上下文注入（隐性专家指令）

为了解决 \&\#34;既要给模型注入大量专家指令，又不能让这些内容刷屏给用户\&\#34; 的矛盾，Skill 采用了**双通道消息机制**：

- **显性消息**：给用户看的简短提示，比如`The \&\#34;pdf\&\#34; skill is loading`，让用户知道系统正在做什么

- **隐性消息**：带有`isMeta: true`标记的隐藏消息，包含了 \[SKILL\.md\]\(SKILL\.md\) 的完整内容（可能几千字），这条消息会发送给 Claude 的 API，进入模型的短期记忆，但用户在界面上完全看不到

这相当于，当你需要处理 PDF 的时候，系统偷偷给 Claude 塞了一本 \&\#34;PDF 处理专家手册\&\#34;，让它瞬间学会了怎么处理这类任务，而你完全不需要看到这本手册的内容。

#### B\. 执行上下文修改（动态权限调整）

Skill 不仅能注入文字指令，还能动态修改模型的执行环境：

- **工具权限**：临时授予特定的工具权限，比如平时不允许 Claude 随便运行 Bash 命令，但加载 PDF Skill 后，会临时允许它使用`pdftotext`这类特定的 Bash 命令

- **模型切换**：部分 Skill 可以强制切换模型，比如简单的文本处理用轻量的 Sonnet，复杂的深度推理切换到 Opus

## 三、Skill 与模型的关系 —— 利用上下文学习的 \&\#34;专家移植\&\#34;

Skill 与大模型的关系，本质上是**利用模型的上下文学习（In\-Context Learning）能力，实现领域专家知识的动态移植**。

### 3\.1 不修改模型，只利用模型的原生能力

Skill 本身不会对模型的权重做任何修改，也不会做微调，它完全依赖模型已经具备的两个核心能力：

1. **语言理解与推理能力**：模型能够根据 Skill 的简短描述，判断用户的需求是否匹配这个 Skill 的能力

2. **上下文学习能力**：当 Skill 的详细指令被注入到上下文后，模型能够快速学习这些指令，按照指令的要求执行任务，就像一个实习生拿到了操作手册，瞬间就能按照规范干活

### 3\.2 模型是 Skill 的决策与执行核心

整个 Skill 的生命周期，模型都是绝对的核心：

1. **技能匹配阶段**：模型自己阅读所有 Skill 的描述，自己判断当前用户的需求应该用哪个 Skill，没有任何外部的意图分类系统帮它做决策

2. **任务执行阶段**：模型根据注入的 Skill 指令，自己拆解任务、规划步骤、调用工具，Skill 只是告诉它 \&\#34;应该怎么做\&\#34;，具体的推理和决策还是模型自己完成

这就像一个全科医生，平时只带一个专家通讯录，遇到心脏问题的时候，他自己判断需要找心脏外科专家，然后专家把全套的手术指南给他，他就能按照指南完成手术，整个过程中，医生的核心能力没有变，只是临时获得了专家的知识。

### 3\.3 动态适配的模型调度

Skill 还可以反过来对模型进行动态适配：

- 针对不同的任务，Skill 可以指定不同的模型，比如简单的文件处理用低成本的轻量模型，复杂的代码生成用高性能的大模型

- 这种调度完全是动态的，不需要用户手动切换，Skill 会根据任务的需求自动调整

## 四、Skill 与 Tool 的关系 —— 元工具与原子能力的协同

Skill 与传统的 Tool（工具）是完全不同层级的概念，二者是协同而非竞争的关系。

### 4\.1 核心差异对比

|维度|传统 Tool|Skill|
|---|---|---|
|**执行模型**|同步直接执行，返回结果|提示词扩展，修改上下文|
|**核心作用**|提供原子的执行能力，比如读文件、写文件、运行命令|提供工作流指导，告诉模型怎么组合使用这些原子能力|
|**返回值**|工具执行的结果数据|对话上下文的修改 \+ 执行环境的修改|
|**定位**|AI 的 \&\#34;手脚\&\#34;，负责具体的操作|AI 的 \&\#34;大脑\&\#34;，负责决策和流程|

### 4\.2 元工具对原子工具的编排

传统的 Tool 是原子能力，比如`Read`、`Write`、`Bash`，这些能力是通用的，但用户需要的不是这些零散的能力，而是 \&\#34;帮我把这个 PDF 里的表格提取出来转成 Excel\&\#34;。

Skill 的作用，就是把这些零散的原子工具编排成一个完整的工作流：

1. 它告诉模型：首先用`Read`工具读取 PDF 文件

2. 然后用`Bash`工具调用`pdftotext`脚本提取表格

3. 然后用`Write`工具把结果写入 Excel 文件

4. 最后把结果整理成用户需要的格式

Skill 本身不会做这些操作，它只是告诉模型，你应该用这些工具，按照这个顺序来做，模型根据这个指导，自己去调用这些工具完成任务。

### 4\.3 动态的工具权限管理

Skill 还解决了工具权限的安全问题：

- 日常状态下，为了安全，系统会限制模型的工具权限，比如不让它随便运行任意的 Bash 命令

- 当加载某个 Skill 的时候，系统会临时放开这个 Skill 需要的特定工具权限，比如 PDF Skill 只会放开`pdftotext`相关的命令权限，用完之后就会收回

- 这样既保证了安全，又让模型能够获得完成任务所需的权限

## 五、OpenClaw 中 Skill 的工程实现

OpenClaw 作为兼容 Claude Skill 生态的开源 AI Agent 框架，完整复刻了 Skill 的核心能力，并做了工程化的落地，实现了 \&\#34;一次开发，多平台复用\&\#34; 的目标。

### 5\.1 Skill 的包结构定义

OpenClaw 中的 Skill 本质上是一个标准化的文件夹，结构完全兼容 Claude 官方规范：

```Plain Text
my-skill/
├── SKILL.md          # 核心定义文件，YAML Frontmatter + Markdown指令
├── scripts/          # 可选的脚本文件，比如处理PDF的Python脚本
├── references/       # 可选的参考文档，比如API文档、规范文档
└── assets/           # 可选的资源文件，比如模板、字体等
```

其中核心的`SKILL\.md`采用**YAML Frontmatter \+ Markdown**的格式，示例如下：

```yaml
---
name: pdf-processor
description: 处理PDF文件，提取表格、文本，转换格式
version: 1.0.0
requires:
  env:
    - PYTHONPATH
  bins:
    - pdftotext
os: ["darwin", "linux"]
emoji: 📄
allowed-tools: ["Bash(pdftotext:*)", "Read", "Write"]
---

# PDF处理专家技能
## 工作流程
1. 首先读取用户提供的PDF文件路径
2. 使用pdftotext工具提取PDF中的文本和表格
3. 如果用户需要转Excel，将表格数据整理成Excel格式
4. 生成处理报告，告知用户处理结果
...
```

### 5\.2 渐进式披露的三层加载机制

OpenClaw 完整实现了 Skill 的渐进式披露机制，最大化节省 Token：

1. **第一层：元数据层**：每次会话启动时加载，仅包含 Skill 的名称、描述、标签，每个 Skill 仅占用不到 100Token，100 个 Skill 也才 5000Token

2. **第二层：指令层**：当模型匹配到需要使用某个 Skill 时，动态加载`SKILL\.md`的完整正文，包含详细的工作流指令

3. **第三层：资源层**：在执行过程中，当需要用到参考文档、脚本的时候，才会按需加载对应的资源文件，哪怕是几百页的 API 文档，不触发就永远不占用上下文

### 5\.3 技能的管理与调度

OpenClaw 实现了灵活的技能管理机制：

#### 四层加载优先级

为了适配不同的使用场景，Skill 采用了四层优先级加载，保证灵活性：

1. **工作区层（最高）**：当前项目的技能，仅当前 Agent 可见，适合项目专属的定制技能

2. **插件层**：通过插件发布的技能，动态加载

3. **用户层**：用户本地的技能，本机所有 Agent 共享

4. **系统层（最低）**：内置的默认技能，随安装包发布

#### 热重载机制

OpenClaw 默认开启 Skill 文件的监听，修改 Skill 文件后，系统会自动热重载，不需要重启服务，开发者可以实时测试修改后的技能，极大提升了开发效率。

### 5\.4 Agent 层的 Skill 执行流程

在 OpenClaw 的 Agent 执行循环中，Skill 的完整执行流程如下：

1. **会话初始化**：Agent 扫描所有 Skill 的元数据，将其整理成列表，注入到`Skill`元工具的描述中

2. **用户请求处理**：用户发送请求后，Agent 将请求和元工具描述一起发送给大模型

3. **模型决策**：模型根据用户请求和 Skill 的描述，判断是否需要调用某个 Skill

4. **Skill 加载**：如果模型决定调用某个 Skill，系统会：

    - 给用户发送 \&\#34;正在加载 XX 技能\&\#34; 的提示

    - 读取`SKILL\.md`的完整内容，生成隐藏的元消息注入到对话上下文

    - 动态修改当前会话的工具权限，放开该 Skill 允许的工具

    - 如果需要，切换到该 Skill 指定的模型

5. **任务执行**：模型带着注入的 Skill 指令，按照工作流的要求，调用工具完成任务

6. **上下文清理**：任务完成后，Skill 的指令和权限会被自动清理，Agent 恢复到通用状态

### 5\.5 安全与兼容性保障

OpenClaw 在实现 Skill 的同时，也做了完善的安全保障：

- **技能安全扫描**：加载 Skill 时，会自动扫描脚本的安全性，防止恶意代码

- **沙箱执行**：Skill 中的脚本会在隔离的沙箱中执行，防止影响宿主系统

- **模型无关兼容**：由于 Skill 本质是提示词，OpenClaw 的 Skill 可以兼容所有支持的大模型，无论是云端的 Claude、GPT，还是本地的开源模型，都能无缝使用

## 总结

Skill 是 Anthropic 在 AI Agent 能力扩展上的一次革命性创新，它从第一性原理出发，利用大模型的上下文学习能力，通过动态提示词注入的方式，实现了低成本、高效率、可移植的能力扩展。它不是 MCP 的替代，而是对 MCP 的补充，二者协同构建了完整的 AI 能力生态。

而 OpenClaw 作为开源实现，将这一能力带到了开源社区，让开发者能够低成本地构建自己的定制化 AI Agent，实现 \&\#34;一次教学，终身复用\&\#34; 的目标，这也是为什么 OpenClaw 能在短短几个月内成为 GitHub 历史上增长最快的项目之一的核心原因。
