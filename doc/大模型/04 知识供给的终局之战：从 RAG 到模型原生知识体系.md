# 知识供给的终局之战：从 RAG 到模型原生知识体系

作者：向春（架构师）
日期：2026-05-19
背景：作为系列第四篇，本文聚焦一个在前三篇中反复触及但未系统展开的核心议题——**知识如何有效地供给模型和智能体**。结合 Claude Code Memory、Karpathy LLM Wiki、Claude Skill 等前沿实践的深度跟踪与穿刺验证，对知识供给的技术路线做系统性分析，并提出一个非共识判断：以 RAG 为代表的"查询时理解"范式正在被"编译时理解"范式取代，模型原生知识体系（LLM-Native Knowledge Infrastructure）才是知识供给的终局方向。

## 一、从一个未竟之题说起

### 1.1 推理时计算的盲区

在第二篇中，我们深入分析了推理时计算——它是当前最关键的"乘数因子"，通过多路径搜索和自我验证将单步正确率推到接近 1。但我们也坦诚地指出了一个盲区：

> 推理时计算不解决私域知识问题。它解决的是"想得深不深"的问题，不是"知道不知道"的问题。

一个推理能力再强的模型，面对它从未见过的 DAIP 配置体系、iDN 部署架构、OKI 运行态模型，也只能"巧妇难为无米之炊"。推理是最好的厨师，但厨师需要食材。

第二篇的 5.3 节列出了五条私域知识接入路径（精调、RAG、GraphRAG、LLM Wiki、大上下文注入、工具调用），但每条只用了一两句话概括。本文的任务，就是把"食材"问题展开为一篇完整的、系统性的分析。

### 1.2 一个被广泛低估的事实

回到第一篇确立的乘法公式：

```
模型能力 = 知识覆盖 × 推理深度 × 指令遵循 × 元认知
```

当前行业的注意力高度集中在"推理深度"（推理时计算、o 系列、extended thinking）和"指令遵循"（RLHF、对齐训练）上。但在乘法公式中，**知识覆盖为零，乘积为零**——推理能力再强也只是在错误的知识基础上深度推理，"想得越深，错得越远"。

更关键的是：在企业级智能体场景中，知识覆盖几乎总是最弱的那个因子。因为通用模型的训练数据不包含你的私域知识——你的配置体系、你的架构决策、你的变更流程、你的历史故障经验。而推理深度、指令遵循、元认知这三项，顶级闭源模型（opus 级别）已经做到了相当高的水准。

这意味着：**对大多数企业而言，知识供给是提升智能体价值的最大杠杆，也是当前最薄弱的环节。**

### 1.3 本文要回答的三个问题

1. 知识供给的本质矛盾是什么？——为什么"把知识喂给模型"这件看似简单的事，工程上这么难做好？
2. 主流方案的深度对比——RAG、精调、大上下文、LLM Wiki、Memory、Skill，各自的机制、优势、根本局限是什么？
3. 终局方向在哪里？——哪种范式更有前景？企业应该把知识基础设施的投入押在哪个方向？

## 二、知识供给的本质矛盾

### 2.1 阻抗不匹配

企业里的知识存在于五种形态中：

| 形态 | 载体 | 示例 |
|:-----|:-----|:-----|
| 人脑中的隐性经验 | 不可直接访问 | "老张知道这个坑千万别踩"、"上次那个故障是因为……" |
| 散乱文档 | Confluence、飞书、邮件、IM 聊天记录 | 方案评审记录、会议纪要、故障复盘 |
| 代码和配置 | Git 仓库、配置中心、Helm Chart | 注释、命名约定、配置层级关系、values.yaml 结构 |
| 流程和规范 | 流程引擎、审批系统、规范文档 | 变更审批流程、发布规范、故障处理手册 |
| 数据和日志 | 数据库、监控系统、日志平台 | 历史操作记录、告警模式、性能基线 |

而模型能消费的知识形态只有一种：**上下文窗口中的文本序列**。

所有知识供给方案，无论名字叫 RAG、精调、Memory 还是 Skill，本质上都在做同一件事——**把各种形态的知识转译成模型能在上下文中有效消费的文本**。

"有效消费"四个字是关键。不是"能看到"就行——128K 窗口塞满了散乱文档，模型"看到了"但未必"理解了"。这就是第一篇中指出的上下文利用有效性问题：信息带宽 ≠ 认知带宽。

### 2.2 三层转译挑战

从原始知识到模型有效消费，需要经过三层转译，每一层都有显著的信息损耗：

```
第一层：显性化 —— 从人脑到文本
  隐性经验、直觉判断 → 可被描述的文字
  损耗：最大。大量隐性知识难以或不愿被表达

第二层：结构化 —— 从散乱文本到有组织的知识
  散乱文档、聊天记录、代码注释 → 有结构、有关联、有层次的知识体
  损耗：中等。结构化过程可能扭曲原意或遗漏关联

第三层：适配化 —— 从结构化知识到模型可消费的格式
  知识体 → 适合模型注意力机制的上下文文本
  损耗：取决于方案。RAG 的 chunk 切割损耗大，Skill 的按需加载损耗小
```

**不同的知识供给方案，本质上是在不同的层、用不同的方式来完成这三层转译。** 方案之间的优劣差异，归根结底是转译质量和转译效率的差异。

### 2.3 一个核心洞察：加工时机决定知识质量

三层转译中最关键的变量，不是"用什么技术"，而是"**谁在什么时候做了理解性加工**"。

| 加工时机 | 含义 | 代表方案 |
|:---------|:-----|:---------|
| 训练时（train-time） | 知识在模型训练阶段被压缩进权重 | 精调、持续预训练 |
| 摄入时（ingest-time） | 知识在进入知识库时被理解、整理、结构化 | LLM Wiki、Skill 编写 |
| 查询时（query-time） | 知识在用户提问时才被检索和临时拼接 | RAG、大上下文注入 |
| 交互时（interaction-time） | 知识在人机交互过程中被逐步积累和修正 | Memory、数据飞轮 |

这四个时机不是简单的"哪个更好"——而是对应着不同类型知识的最优加工策略。后面会详细展开这个映射关系。但有一个总体判断可以先给出：

> **加工越前置，运行时的知识质量越高、认知负担越低。** RAG 把最重的理解负担放在查询时（模型拿到碎片后临时理解），LLM Wiki 把理解前置到摄入时（知识进库时就已被理解和组织），Memory 更进一步——把理解分散到每一次交互中持续积累。

## 三、主流方案的深度剖析

### 3.1 RAG：查询时理解的结构性缺陷

RAG（Retrieval-Augmented Generation）是当前行业最主流的知识供给方案。它的架构直觉上很合理：用户提问 → 从知识库检索相关文档 → 把检索结果塞入上下文 → 模型基于这些信息回答。

但在深度使用中，RAG 暴露了三个结构性缺陷：

**缺陷一：检索环节由最弱的组件执行。**

RAG 的检索通常依赖 Embedding 模型做语义相似度匹配。但 Embedding 模型的语义理解能力远弱于生成模型——它能判断"两段文字在表面上相似"，但不能判断"这段知识在逻辑上与当前问题相关"。

举一个具体的例子：用户问"修改 HDFS 副本数配置会影响正在运行的写入任务吗？"——RAG 检索到了 HDFS 副本配置的文档（表面相关），但没有检索到 HDFS Pipeline 写入机制的文档（深层相关）。因为后者的 Embedding 向量与查询的余弦相似度不够高——虽然它是回答这个问题的关键知识。

这是 RAG 的系统性风险：**最关键的知识恰恰可能是与查询表面相似度最低的那些。** 深层关联需要推理才能发现，而 Embedding 模型不做推理。

**缺陷二：Chunk 切割破坏知识结构。**

RAG 需要把文档切成固定大小的 chunk 才能建索引。但知识不是均匀分布在文本中的——一个完整的论证可能跨越 3 个段落，一个配置项的含义可能依赖前面 5 页的上下文。chunk 切割把这些完整的知识单元打碎了，模型拿到的是碎片，不是知识。

更隐蔽的问题是：chunk 之间的关联被完全丢失了。文档 A 的第三章和文档 B 的附录二之间可能有关键的逻辑依赖，但在 chunk 化之后这种依赖不再存在——因为它们变成了向量空间中两个独立的点。

**缺陷三：查询时理解的认知负担过重。**

用 p^n 公式来分析 RAG 的完整链条：

```
用户查询 → Embedding 编码(p₁) → 向量检索(p₂) → Rerank(p₃) → 上下文拼接(p₄) → 模型理解碎片(p₅) → 模型推理(p₆)
```

六个乘法环节。其中 p₁~p₃ 由 Embedding 模型和检索系统完成（能力受限），p₅ 要求生成模型在拿到碎片后"临时重建完整的知识图景"——这个步骤的认知负担极重，而且每次查询都要重复。

对比一下人类专家的工作方式：一个经验丰富的架构师不会每次被问到 HDFS 配置问题时都重新翻文档——他脑中有一个组织好的 HDFS 知识体系，被问到时直接从这个体系中调取相关部分。**RAG 让模型每次都"重新翻文档"，而不是让模型拥有一个组织好的知识体系。**

### 3.2 GraphRAG：正确的直觉，错误的执行路径

GraphRAG 试图解决 RAG 的"碎片化"问题——通过构建知识图谱来保留实体间的关系，用图检索替代向量检索来支持多跳推理。直觉上，这是正确的进化方向。

但如第二篇所述，GraphRAG 恰好踩在大模型最弱的环节上——自动实体抽取和关系抽取。

问题链条是这样的：

```
自动抽取实体（准确率 ~80%）
  → 自动识别关系（准确率 ~70%）
    → 入库（错误知识也一起入了）
      → 图检索（在错误的图上做推理）
        → 生成答案（基于错误的推理）
```

每一步的错误都会传导到下一步。80% × 70% = 56%——还没到推理环节，知识基础的可靠性就已经跌到了一半。在 p^n 的框架中，这是灾难性的起点。

更深层的问题是：**GraphRAG 试图让机器自动完成"理解知识间的关系"这件事，但这件事的本质是高质量推理——恰恰是当前自动化最不可靠的环节。** 用 80% 准确率的自动抽取去构建一个知识图谱，不如用人类专家花几小时写一份结构化的知识文档。

这也是为什么在第二篇中我们提出了"本体增强提示（Schema-Guided Prompt）"可能是更务实的方向——放弃"机器自动抽关系"，改用"人定框架 + 模型遵循"。后面会看到，LLM Wiki 和 Skill 正是这条路线的具体实现。

### 3.3 精调：知识的"一次性固化"

精调解决的是"让模型在训练阶段记住特定领域的模式"。LoRA/QLoRA 大幅降低了成本，使之成为一种实用的工程手段。

但精调在知识供给层面有三个根本局限：

**局限一：知识冻结。** 训练完成的那一刻，知识就停止更新了。你的配置体系在演化、你的架构在调整、你的流程在迭代——但精调后的模型还停留在训练时的认知。要跟上变化就得反复精调，每次精调都是一次成本支出。

**局限二：无法精调最强的模型。** 在第一篇中，我们确立了一个核心认知——智能体场景需要最强的推理模型。但 opus 级别的闭源模型不开放精调。你只能精调次一级的模型，这意味着为了注入知识而牺牲了推理能力——在乘法公式中，这是一个不利的交换。

**局限三：精调擅长的不是"事实性知识"，而是"认知模式"。** 精调的真正价值在于让模型习得一种"思考方式"——比如你们团队写架构文档的风格、评审代码时关注的维度、做技术决策时的权衡偏好。这类"方法论知识"变化慢、难以通过上下文注入传递，是精调的甜蜜区。但具体的配置项、具体的 API 参数、具体的变更流程——这些"事实性知识"变化快、需要精确，用精调来注入是对工具的误用。

### 3.4 大上下文注入：简单但有天花板

直接把完整文档塞入上下文窗口——不切割、不检索、不建索引。Cursor 的成功很大程度上受益于此——直接把大量代码文件塞进上下文，比设计精巧的代码检索管道更可靠。

优势显而易见：没有中间环节，没有信息损耗，模型看到的是完整的原始材料。在 p^n 的框架中，链条极短：

```
用户查询 → 相关文档直接注入上下文(p₁) → 模型理解和推理(p₂)
```

两步。这解释了为什么在很多场景下，"简单粗暴地塞进去"比"精巧的 RAG 管道"效果更好——更少的环节意味着更少的失败点。

但天花板也很清晰：

**天花板一：窗口终究有限。** 即使 Gemini 推到了 1M+ token，一个中等规模企业的完整知识体量也远超这个窗口。你不可能把所有知识都塞进去，选择"塞什么"本身就是一个需要智能判断的问题。

**天花板二：信息淹没效应。** 第一篇中已经指出"Lost in the Middle"问题——模型对上下文中间位置的信息注意力显著衰减。10M token 的窗口塞满了文档，关键信息在第 5M 位置，大概率被"淹没"。**窗口越大，信息淹没的风险越高。**

**天花板三：成本线性增长。** 每一个注入上下文的 token 都要消耗推理算力。在 Agent 场景中，一个会话可能有几十轮交互，每轮都携带完整的上下文——成本会迅速失控。

大上下文注入是一种"够用就好"的策略：当你知道当前任务需要哪些具体文档时，直接塞进去，效果确实好。但它不是一种可规模化的知识管理方案——它缺乏组织、缺乏持续性、缺乏积累机制。

### 3.5 工具调用：实时但依赖推理

模型通过调用 API、查数据库、读文件系统来实时获取知识。这条路径的独特价值在于时效性——它获取的是"此时此刻"的状态，不是任何缓存或快照。

但在第一篇中我们已经分析过：工具调用的质量严格依赖第二层能力（工具交互能力）和第一层能力（推理能力）。模型得"想清楚该调什么工具、传什么参数、怎么解读返回结果"——这本身就需要推理能力和领域知识的支撑。

一个典型的困境：模型需要知道"应该查哪个数据库的哪张表才能找到当前 HDFS 集群的副本配置"——但这个知识本身就是私域知识。工具调用解决的是"最后一公里的获取"，不是"知道去哪获取"。后者仍然需要其他知识供给方式来覆盖。

### 3.6 小结：各方案的乘法链条长度对比

用 p^n 框架统一对比各方案的链条长度（越短越可靠）：

| 方案 | 链条 | 环节数 | 最弱环节 |
|:-----|:-----|:------|:---------|
| RAG | 查询编码 → 向量检索 → Rerank → 拼接 → 理解碎片 → 推理 | 6 | 检索（Embedding 能力弱） |
| GraphRAG | 查询理解 → 图检索 → 子图提取 → 拼接 → 推理 | 5 | 图的准确性（自动抽取不可靠） |
| 精调 | 直接推理（知识已在权重中） | 1 | 知识时效性（冻结的） |
| 大上下文 | 文档注入 → 推理 | 2 | 信息淹没 |
| 工具调用 | 判断调什么 → 构造参数 → 执行 → 解读结果 → 推理 | 5 | 判断和参数构造（需要元知识） |

这张表遗漏了三个方案：LLM Wiki、Memory、Skill。它们恰好是我认为代表终局方向的方案——下一章展开。

## 四、范式转移：从"查询时理解"到"编译时理解"

### 4.1 一个类比

程序语言有两种执行方式：解释执行和编译执行。

**解释执行**：每次运行时，逐行翻译、逐行执行。开发快，但每次运行都要重复翻译的工作，效率低。

**编译执行**：运行前先把源代码编译成机器码。编译阶段投入一次性成本，但运行时直接执行编译产物，效率高得多。

RAG 就是知识的"解释执行"——每次查询时，从原始文档中临时检索、临时拼接、临时理解。LLM Wiki 是知识的"编译执行"——摄入时把原始材料一次性"编译"成结构化的知识产物，查询时直接使用编译结果。

从程序语言的历史来看，编译执行在性能敏感的场景中几乎总是胜出。知识供给可能正在经历同样的转变。

### 4.2 LLM Wiki：知识的编译器

Karpathy 在 2026 年 4 月提出的 LLM Wiki 架构，是"编译时理解"范式的最清晰表达。

**三层架构：**

```
raw/        ── 原始材料层（不可变的事实源）
                论文、代码、网页、数据集、会议纪要……
                一切未经加工的原始信息

wiki/       ── 编译产物层（结构化知识）
                百科式概念页、实体页、关系页、摘要页
                带交叉引用和反向链接
                每一个知识点都可追溯到 raw/ 中的原始来源

CLAUDE.md   ── Schema 层（编译规则）
                定义 wiki 的结构标准、分类体系、质量要求
                相当于编译器的"语法规则"
```

**三阶段操作：**

**阶段一：摄入（Ingest）。** 把原始材料收集到 `raw/` 目录。不做任何加工，保留原始面貌。关键原则是"原始材料不可变"——它是事实的源头，编译产物可以重新生成，但原始材料不能丢。

**阶段二：编译（Compile）。** LLM 读取原始材料，按照 Schema 的要求，生成结构化的 wiki 页面。这不是简单的"摘要"——而是理解、关联、去重、整合。一份 50 页的技术白皮书，可能被编译成 5 个概念页、3 个实体页和若干交叉引用。编译过程中，LLM 需要做大量的推理工作——识别核心概念、判断概念间的关系、决定信息的层级结构。**这正是"在摄入时完成理解"的核心含义。**

**阶段三：维护（Lint）。** LLM 定期扫描整个 wiki，执行"健康检查"——发现不一致、补充缺失的交叉引用、识别新的关联、标记过时的信息。这让知识库成为一个**自我修复的系统**，质量随时间推移而提升，而不是像传统文档那样"写完即腐烂"。

**为什么这比 RAG 好？**

回到 p^n 框架：

```
RAG 查询链条：
  查询编码(p₁) → 向量检索(p₂) → Rerank(p₃) → 上下文拼接(p₄) → 理解碎片(p₅) → 推理(p₆)
  6 个环节，其中 p₁~p₃ 由 Embedding 模型执行（弱）

LLM Wiki 查询链条：
  模型判断需要哪些知识页(p₁) → 加载知识页(p₂) → 推理(p₃)
  3 个环节，其中 p₁ 由强模型执行
```

链条从 6 步缩短到 3 步。而且最关键的"检索"环节（p₁），从 Embedding 模型的向量匹配升级为了强模型的语义推理。这两个改进叠加——更短的链条 + 更强的单步能力——在乘法公式中的效果是指数级的。

更根本的改进是：RAG 的模型拿到的是"碎片化的原始材料"，每次查询都从零开始理解；LLM Wiki 的模型拿到的是"已被理解、结构化、关联过的知识成品"。**同一个模型，输入质量的差异直接决定了输出质量的差异。**

### 4.3 Claude Code Memory：交互中持续积累的知识飞轮

如果说 LLM Wiki 是知识的"编译器"，那 Claude Code 的 Memory 系统就是知识的"增量学习机"。

Claude Code 在 2026 年 2 月推出的 Auto-Memory 系统是目前工程上最成熟的 Agent Memory 实现。它的架构设计蕴含了几个深刻的决策。

**两层索引 + 按需加载：**

```
┌──────────────────────────────────────────────────────────┐
│  Layer 1: MEMORY.md（永远加载，≤200 行 / ~25KB）           │
│  ─ 浓缩的经验教训，带日期戳                                │
│  ─ 近期会话的核心摘要                                     │
│  ─ 路由表：将领域映射到 Layer 2 的具体 topic 文件           │
│  ─ 每轮都消耗 token，但极度精简                           │
├──────────────────────────────────────────────────────────┤
│  Layer 2: Topic Files（按需加载，每轮最多 5 个）            │
│  ─ user_role.md / debugging.md / project_auth.md ……      │
│  ─ 模型先 ls() 看有哪些文件                               │
│  ─ 根据文件名语义 + 当前任务，推理判断加载哪几个              │
│  ─ 每个文件聚焦一个领域，内容密度高                         │
├──────────────────────────────────────────────────────────┤
│  Layer 3: Session Transcripts（极少访问）                  │
│  ─ JSONL 格式的完整会话记录                                │
│  ─ 仅通过 grep 做针对性关键词搜索                          │
│  ─ 当 Layer 1&2 不够时的最后手段                           │
└──────────────────────────────────────────────────────────┘
```

**关键设计决策一：用 LLM 推理替代向量检索。**

Claude Code 的设计者明确拒绝了 RAG 来做 Agent Memory，原话很精辟：

> "向量数据库解决的是一个错误的问题。Agent 的记忆不是知识库——它是小的、频繁变化的、必须人类可读可编辑的观察笔记。"

模型通过 `ls()` 列出 topic 文件清单，根据文件名的语义含义推理判断当前任务需要哪些文件，然后 `read_file()` 加载选中的文件。整个"检索"过程完全由强模型的推理来驱动，没有 Embedding、没有向量索引、没有相似度计算。

这个决策的底层逻辑与第一篇的框架完全一致：**当模型推理能力足够强时，让模型自己做检索决策，比用一个弱得多的 Embedding 模型做检索，端到端效果更好。** 这就是"less framework, more model"在知识供给领域的具体体现。

**关键设计决策二：Markdown 为唯一存储格式。**

没有数据库、没有向量索引、没有二进制格式——所有记忆都是 `.md` 文件。

这看似"原始"，实际上是一个深思熟虑的选择：
- 人类可以直接阅读、编辑、审校——知识的最终所有权属于人
- 可以用 git 做版本管理——每一次修改都有迹可查
- 模型原生擅长理解 Markdown——不需要额外的解析层
- 极低的技术门槛——任何人都可以参与知识维护

**关键设计决策三：四种语义类型。**

Memory 将知识分为四种类型：

```
User      ── 用户是谁（角色、偏好、技术背景）
Feedback  ── 怎么行为（用户纠正过的错误模式、确认过的正确模式）
Project   ── 在做什么（项目状态、当前进展、待解决的问题）
Reference ── 去哪找（关键文件路径、架构决策记录、重要入口点）
```

这四种类型恰好覆盖了第一篇中意图对齐的四个层次——

| Memory 类型 | 对应意图对齐层次 | 解决的问题 |
|:-----------|:--------------|:---------|
| User | 字面对齐 | 模型知道"你是谁"，不会答非所问 |
| Project | 目标对齐 | 模型知道"你在做什么"，理解请求的上下文 |
| Feedback | 约束对齐 | 模型知道"你没说但隐含的要求"，避免重复犯错 |
| Reference | 执行所需知识 | 模型知道"去哪找信息"，降低工具调用的不确定性 |

**Memory 最独特的价值：将隐性知识自动显性化。**

这是 Memory 区别于所有其他方案的核心特征。RAG 需要知识先被写成文档，LLM Wiki 需要原始材料先被收集，Skill 需要专家先编写工作流——它们都需要一个显式的"知识输入"步骤。

但 Memory 的知识来源是**人机交互过程本身**。每次你纠正模型的错误（"不，这个配置不是在 values.yaml 里改，是在 configmap-override 里"），Memory 就记下了一条你脑中的隐性知识。你不需要专门坐下来"写文档"，纠正行为本身就是知识输入。

这解决了知识供给的最大瓶颈——第一层转译（从人脑到文本）。Memory 让这个转译不再需要专门的投入，而是发生在日常工作的自然流程中。

### 4.4 Claude Skill：方法论知识的最优载体

第三篇已经详细分析了 Skill 的技术架构，这里从知识供给的视角做一个重新定位。

Skill 本质上是一种知识供给方式——只是它供给的不是"事实性知识"（是什么），而是**"方法论知识"（怎么做）**。

一个 PDF 处理 Skill 包含的不是"PDF 格式的技术规范"，而是"处理 PDF 文件的标准流程"——先用什么工具、按什么顺序、遇到什么情况做什么处理。这是一种流程性知识、规范性知识、经验性知识的封装。

Skill 的渐进式披露机制，与 Claude Memory 的两层索引在架构思想上高度一致：

| 层级 | Claude Memory | Claude Skill |
|:-----|:-------------|:-------------|
| 始终加载 | MEMORY.md（≤200 行摘要） | 元工具描述（所有 Skill 的名称 + 100 字简介，~5000 token） |
| 按需加载 | Topic Files（模型推理选择） | SKILL.md 完整内容（模型匹配后加载） |
| 极少使用 | Session Transcripts（grep 搜索） | scripts / references / assets（执行时才读取） |

这不是巧合。**渐进式披露是"模型原生知识体系"的共同设计原则。** 它的底层逻辑是：上下文窗口是稀缺资源，不应该被不相关的信息占据。先用极低成本让模型"知道有什么可用"（摘要/索引），再让模型自己决定"需要详细了解什么"（按需加载），最后只在执行时才加载最具体的资源。

### 4.5 三者的统一：模型原生知识体系

Claude Skill、Claude Memory、Karpathy LLM Wiki——看似是三个独立的技术，但它们共享一组深层的设计原则：

| 共同原则 | 说明 | 与 RAG 的对比 |
|:---------|:-----|:-------------|
| Markdown 为唯一介质 | 人和模型都能直接读写的纯文本 | RAG 依赖向量数据库，人不可读 |
| LLM 推理做检索 | 模型自己看文件名/描述来决定加载什么 | RAG 用 Embedding 做相似度匹配 |
| 渐进式披露 | 摘要先行，细节按需，资源最后 | RAG 每次检索固定数量的 chunk |
| 人机共同维护 | 人定规则/框架，LLM 执行/积累，人审校 | RAG 的知识库通常是"倒入即不管" |
| 编译时理解 | 知识在摄入时已被理解和组织 | RAG 在查询时才临时理解碎片 |
| 持续积累的飞轮 | 越用越好——Memory 记录纠正，Wiki 补充知识，Skill 沉淀流程 | RAG 不具备内生的质量提升机制 |

我将这组共同原则统称为**"模型原生知识体系"（LLM-Native Knowledge Infrastructure）**。

"模型原生"的含义是：这些方案的设计完全围绕模型的认知特点来组织知识——模型擅长理解 Markdown（所以用 Markdown），模型擅长推理（所以用推理做检索），模型受注意力机制约束（所以用渐进式披露），模型的上下文学习能力强（所以用动态注入而非固化到权重中）。

与之对比，RAG 的设计是围绕**数据库**的范式来组织知识——切块、编码、建索引、检索、拼接——这是一个数据工程的思路，不是一个认知工程的思路。**它把模型当成一个"需要被喂数据的消费者"，而不是一个"能自主组织和获取知识的认知主体"。**

## 五、知识类型与供给策略的映射

不是所有知识都适合用同一种方式供给。知识类型的差异决定了最优的供给路径。

### 5.1 知识类型分类框架

按两个维度分类——**稳定性**（变化频率）和**形式化程度**（结构化程度）：

```
                    形式化程度 高
                          ↑
        ┌─────────────────┼─────────────────┐
        │  API Schema      │  实时集群状态     │
        │  协议规范         │  当前配置值       │
        │  数据模型定义      │  运行中任务列表   │
        │                  │                 │
        │  最优：Skill/精调  │  最优：工具调用    │
    稳  ├─────────────────┼─────────────────┤ 易
    定  │  团队编码规范      │  项目迭代上下文   │
        │  架构设计原则      │  近期决策和变更   │
        │  评审关注维度      │  当前冲刺目标     │
        │                  │                 │
        │  最优：Skill+Memory│ 最优：Memory      │
        ├─────────────────┼─────────────────┤ 变
        │  领域术语体系      │  历史故障案例     │
        │  行业 know-how   │  运维经验日志     │
        │  技术选型理由      │  排障过程记录     │
        │                  │                 │
        │  最优：LLM Wiki   │  最优：LLM Wiki+RAG│
        ├─────────────────┼─────────────────┤
        │  "这个坑千万      │  "上次处理类似    │
        │   别踩"的经验     │   问题的过程"     │
        │  隐性判断直觉      │  未文档化的操作   │
        │                  │                 │
        │  最优：先显性化    │  最优：Memory     │
        │  再 Skill/Wiki   │  （交互中自然积累）│
        └─────────────────┴─────────────────┘
                          ↓
                    形式化程度 低
```

### 5.2 每个象限的供给策略

**象限一（稳定 × 高形式化）：API 规范、协议定义、数据模型。** 变化极慢，结构清晰。最适合封装为 Skill（以标准流程指导模型如何使用这些规范）或在基座模型精调中固化。

**象限二（易变 × 高形式化）：实时集群状态、当前配置值。** 变化快但结构清晰。只有工具调用能满足时效性要求——模型通过 API 查询当前状态，获取的是"此时此刻"的事实。

**象限三（稳定 × 中等形式化）：团队规范、架构原则。** 变化慢但不如 API 规范那么严格形式化。Skill 负责封装标准流程（"怎么做"），Memory 的 Feedback 类型记录团队特有的偏好和约束（"不要这样做"）。

**象限四（易变 × 中等形式化）：项目上下文、近期决策。** 每天都在变，但有一定结构。这是 Memory 的核心领地——通过交互持续更新 Project 类型的记忆，让模型始终知道"当前在做什么"。

**象限五（稳定 × 低形式化）：领域术语、行业 know-how。** 变化慢但分散在各处。LLM Wiki 最适合——把散乱的领域知识编译成结构化的百科页面，一次编译、长期受益。

**象限六（易变 × 低形式化）：历史故障、运维经验。** 变化相对频繁且分散。LLM Wiki 做基础的结构化整理，对于近期的高频更新部分，可以辅以 RAG 做增量覆盖。

**象限七&八（低形式化底部）：纯隐性经验。** 这是最难处理的知识——它还在人脑中，没有被写下来。对于稳定的隐性经验，需要人主动"显性化"（写成 Skill 或 Wiki 页面）。对于易变的隐性经验，Memory 的纠正机制是最自然的显性化通道。

### 5.3 分层架构：不是选一种，而是组合

上面的分析指向一个结论：**知识供给不存在"一种方案通吃"的银弹，终局是分层架构。**

```
┌──────────────────────────────────────────────────────────┐
│  Skill 层：方法论知识（"怎么做"）                           │
│  ─ 人类专家编写标准流程 + 规范约束                          │
│  ─ 模型按需加载，渐进式披露                                │
│  ─ 适合：稳定的、可标准化的流程性知识                       │
├──────────────────────────────────────────────────────────┤
│  Memory 层：交互积累的知识（"经验/偏好/纠正"）              │
│  ─ 人机协同维护，自然交互中持续积累                         │
│  ─ 两层索引 + 按需加载                                    │
│  ─ 适合：个性化偏好、项目上下文、纠正过的错误模式            │
├──────────────────────────────────────────────────────────┤
│  Wiki 层：领域知识（"是什么/为什么"）                       │
│  ─ LLM 编译 + 人审校，定期 Lint 维护                      │
│  ─ 适合：稳定的领域知识、历史案例、技术决策记录              │
├──────────────────────────────────────────────────────────┤
│  工具层：实时状态（"此时此刻"）                             │
│  ─ API/数据库/文件系统，按需调用                            │
│  ─ 适合：实时数据、动态状态、结构化查询                     │
└──────────────────────────────────────────────────────────┘
```

四层各司其职，互不替代。Agent 在执行一个复杂任务时，可能同时需要四层的知识：

- 从 **Skill** 获取"这类任务的标准处理流程"
- 从 **Memory** 获取"上次做类似任务时用户纠正过的要点"
- 从 **Wiki** 获取"相关组件的架构设计和技术决策背景"
- 从 **工具** 获取"当前集群的实时运行状态"

**RAG 在这个架构中的位置**：它不再是一个独立的知识供给范式，而是退化为 Wiki 层和工具层的一个实现细节——当 Wiki 的规模超过单次加载能力时，用检索做预筛选；当工具调用返回大量结果时，用 Rerank 做过滤。它从"架构的中心"变成了"架构的辅助组件"。

## 六、效果差异的根因分析：为什么 Memory 和 Wiki 比 RAG 好得多

### 6.1 从信息论角度

信息论的核心概念是"信噪比"——有用信息与无关信息的比例。

**RAG 的信噪比问题**：检索返回的 chunk 中，真正与当前问题相关的信息可能只占 30%~50%——其余是切割带来的冗余上下文、部分相关但不关键的段落、表面相似但语义不同的内容。模型需要在这些"噪音"中筛选信号，这消耗了宝贵的推理资源。

**LLM Wiki 的信噪比优势**：Wiki 页面是针对特定概念/实体编写的——每一页都是高度聚焦的知识单元。当模型加载一个 Wiki 页面时，页面中的信息几乎 100% 与当前主题相关。信噪比显著高于 RAG 的检索结果。

**Memory 的信噪比优势**：Memory 记录的是经过交互验证的知识——每一条都是用户确认过的事实、纠正过的错误、明确表达的偏好。没有推测、没有冗余、没有噪音。它可能是所有知识供给方式中信噪比最高的。

### 6.2 从认知负担角度

回到第一篇的上下文利用有效性分析——模型消费知识需要三个递进的能力：定位、判断相关性、正确整合。

**RAG 把三层负担都压在了查询时**：模型拿到 5 个 chunk，需要在这些碎片中定位关键信息、判断哪些与当前问题真正相关、把不同 chunk 中的信息正确关联和整合。这三件事同时做，认知负担极重。

**Wiki 把前两层前置到了编译时**：Wiki 页面在编写时就已经完成了定位（从原始材料中提取关键概念）和相关性判断（按主题组织）。模型在查询时只需要做第三层——正确整合——而且整合的对象是已经结构化的知识页，不是散乱的碎片。

**Memory 把三层都前置到了交互时**：每条 Memory 都已经是经过定位、判断和整合的知识成品。模型在查询时几乎零额外认知负担——直接使用。

### 6.3 从知识的"活性"角度

"活的"知识会随着使用而变得更好，"死的"知识放在那里只会逐渐腐烂。

**RAG 的知识是"死的"**：文档入库后，除非人工维护，否则不会自动更新、不会自动发现不一致、不会自动补充缺失。随着业务演进，RAG 知识库中过时信息的比例只会越来越高——知识腐烂。

**Wiki 的知识是"半活的"**：Lint 机制让 Wiki 具备了自我修复能力——定期扫描不一致、补充交叉引用。但触发 Lint 仍需要外部调度，不是完全自主的。

**Memory 的知识是"活的"**：每一次人机交互都可能产生新的记忆或修正旧的记忆。知识库在使用中自然生长——你不需要专门投入时间"维护"它，使用本身就是维护。这就是第一篇中"数据飞轮"的具体机制：每一次纠正 → Memory 记录 → 下次交互质量提升 → 减少纠正需求 → 但当遇到新问题时产生新的纠正 → 继续积累。

---

## 七、三个非共识判断

### 判断一：知识供给的瓶颈不是技术，而是"知识的显性化"

所有知识供给方案——无论 RAG、Wiki 还是 Memory——都有一个共同的前提：知识必须先以某种形式"存在"于模型能触达的地方。

但大多数企业最有价值的知识**还停留在人脑中**。"老张知道这个坑"、"上次那个故障王工处理了三天才搞定"、"这个配置改的时候一定要先通知运维组"——这些经验是企业最稀缺的知识资产，但它们从未被写下来。

RAG 需要文档，Wiki 需要原始材料，Skill 需要专家编写——它们都需要知识先被"写出来"。**真正的瓶颈不是"怎么检索"或"怎么注入"，而是"怎么让隐性知识变成文字"。**

这正是 Memory 机制最独特的价值所在：它通过**记录人机交互过程中的纠正**，将隐性知识显性化的过程嵌入到了日常工作流中。你不需要专门"写文档"，只需要在 Agent 犯错时纠正它——这个纠正行为本身就是知识的输入。

**行动含义**：与其投入大量资源建设 RAG 管道（解决的是已有文档的检索问题），不如优先建设 Memory 机制（解决的是知识从无到有的问题）。前者优化的是"已有知识的利用效率"，后者提升的是"知识资产的总量"。在乘法公式中，当"知识覆盖"接近零时，提升"检索效率"毫无意义。

### 判断二：Skill 是方法论知识的终极封装，Memory 是事实性知识的自然积累，二者互补构成完整的知识体系

```
            知识类型
┌──────────────────────────────────────┐
│  "怎么做"（方法论、流程、规范）        │ → Skill
│  "是什么"（事实、状态、关系）          │ → Wiki + Memory
│  "为什么"（决策原因、权衡、取舍）       │ → Memory（纠正中积累）+ Wiki（主动整理）
└──────────────────────────────────────┘
```

只有 Skill 没有 Memory，模型知道标准流程但不知道你的具体情况——"按照规范应该这么做，但你们的环境有什么特殊约束我不知道"。

只有 Memory 没有 Skill，模型知道你的情况但不知道最佳实践——"我记得你上次修改配置出了问题，但我不确定正确的操作流程是什么"。

二者组合，模型既掌握了标准流程（Skill），又了解你的特定约束和偏好（Memory），还能查到领域背景知识（Wiki），需要实时数据时还能通过工具获取。这才是一个完整的知识体系。

### 判断三："LLM 推理替代向量检索"是一个正在发生的范式转移

Claude Code 拒绝向量数据库，用 `ls()` + 文件名语义推理来"检索"记忆。Karpathy 的 LLM Wiki 不用 Embedding，用 Markdown 文件的人类可读标题做索引。Claude Skill 用模型自身的语义理解来匹配技能描述——"没有任何硬编码的路由、分类器或正则匹配，所有决策完全发生在 Transformer 的前向传播中"。

三个独立的系统，不约而同地做了同一个选择：**让强模型自己判断需要什么知识，而不是用弱模型（Embedding）做向量匹配。**

这不是偶然。它背后的逻辑链条是这样的：

1. 知识检索的本质是一个**理解问题**——"这条知识与当前问题相关吗？"
2. 理解需要**推理能力**
3. Embedding 模型的推理能力远弱于生成模型
4. 当生成模型的推理能力足够强时，让它自己做检索决策，端到端效果优于 Embedding 检索
5. 随着模型推理能力持续提升，这个优势只会越来越大

这与第一篇的核心主张完全一致："less framework, more model"——当模型足够强时，把决策权交给模型，比交给中间框架效果更好。知识检索只是这个原则在知识供给领域的又一个实例。

**这意味着 RAG 的核心组件——向量索引和 Embedding 检索——的长期价值在缩小。** 未来的知识供给系统，检索层不再是"技术壁垒"，知识的组织质量才是。向量数据库的角色会退化为"大规模知识库的预过滤工具"——当知识量太大、模型不可能逐一审视时，用向量检索先粗筛一遍，再交给模型精判。它从"架构的核心"变成了"架构的性能优化层"。

## 八、建设路径：从零到一的务实建议

### 8.1 评估你的短板

第一步不是选技术方案，而是诚实地评估你的知识资产状况：

| 如果你的短板是…… | 优先投入 | 理由 |
|:----------------|:---------|:-----|
| 知识还在人脑中，几乎没有文档化 | Memory 机制 + 知识显性化运动 | 先解决"从无到有"，再谈"如何组织" |
| 文档很多但散乱、过时、难以利用 | LLM Wiki（编译 + Lint） | 将散乱文档编译为结构化知识 |
| 有标准流程但 Agent 不遵循 | Skill 封装 | 把流程知识转化为 Agent 可消费的格式 |
| 知识体系完善但模型推理不足 | 选择更强的基座模型 + 推理时计算 | 知识够了，提升"厨师"水平 |
| 需要实时数据但 Agent 不知道去哪取 | 工具链建设 + Reference 类 Memory | 建设"最后一公里"的获取通道 |

### 8.2 分阶段建设路径

**第一阶段（立竿见影）：Skill + Memory**

投入最小、见效最快。

- 为你最高频的 5~10 个工作场景编写 Skill。一个 Skill 就是一个 Markdown 文件，写清楚流程、约束、注意事项。5 分钟一个，投入可忽略不计。
- 启用 Memory 机制（Claude Code 的 Auto-Memory，或在其他平台上用 AGENTS.md / .cursorrules 模拟）。开始在交互中积累纠正。
- 效果预期：在你最常用的场景中，Agent 的首次正确率显著提升——因为它不再"从零开始猜"，而是有了标准流程和你的偏好记录。

**第二阶段（夯实基础）：LLM Wiki**

- 把现有的散乱文档收集到 `raw/` 目录。不需要整理——保留原始面貌。
- 编写 Schema（定义 wiki 的结构标准——分类体系、命名规范、质量要求）。
- 用强模型把原始材料"编译"成结构化的 Wiki 页面。人工审校关键页面。
- 建立 Lint 机制——定期让模型扫描 Wiki 的一致性和完整性。
- 效果预期：Agent 在需要领域背景知识时，不再"编造"或给出"教科书式的通用回答"，而是基于你的实际技术栈和业务上下文。

**第三阶段（打通闭环）：工具链 + 飞轮**

- 建设工具调用通道——让 Agent 能查询配置中心、监控系统、数据库。
- 在 Wiki 的 Reference 页面中记录工具的入口和用法，降低 Agent 的工具调用门槛。
- 建立从 Memory 到 Wiki 的沉淀机制——当某条 Memory 被反复引用时，考虑将它固化为 Wiki 页面或 Skill 步骤。
- 效果预期：Agent 具备了完整的知识获取链路——从标准流程（Skill）到领域知识（Wiki）到实时状态（工具）到个性化偏好（Memory），形成闭环。

### 8.3 投入产出比的排序

| 方案 | 建设成本 | 维护成本 | 见效速度 | 长期价值 | 推荐优先级 |
|:-----|:---------|:---------|:---------|:---------|:---------|
| Memory 机制 | 极低（开启即用） | 极低（自然积累） | 快（几次交互后可见） | 高（持续积累的飞轮） | P0 |
| Skill 编写 | 低（Markdown 文件） | 低（按需更新） | 快（写完即用） | 高（可复用、可共享） | P0 |
| LLM Wiki | 中（需要 Schema 设计 + 编译 + 审校） | 中（Lint 可自动化，但需要人审校） | 中（需要一定积累量） | 很高（越积累越有价值） | P1 |
| 工具链 | 高（需要 API 开发） | 中（随系统演进） | 慢（需要基础设施） | 高（实时性不可替代） | P2 |
| RAG 管道 | 高（Embedding + 向量库 + 检索 + Rerank） | 高（知识库需要持续维护） | 中 | 中（正在被新范式侵蚀） | P3 |
| 精调 | 很高（数据准备 + 训练 + 评估） | 很高（每次更新都要重新训练） | 慢 | 低（护城河快速贬值） | P4 |

## 九、收尾：知识基础设施才是真正的护城河

在第一篇中，我们给出了一个核心判断：

> 壁垒不在谁拥有最强的模型，而在谁拥有最深的场景理解、最精准的评测体系、和最扎实的数据飞轮。

现在可以更精确地表述这个判断了：

> 壁垒不在谁的 RAG 管道更精巧，而在谁的知识体系更扎实。

模型会持续商品化。RAG 的工程管道会变成标准组件。但你积累的 Skill 库（编码了你团队的最佳实践）、你的 Memory 体系（沉淀了你的私域经验）、你的 Wiki（编译了你的领域知识）——这些不会被商品化，因为它们的内容只属于你。

知识供给的终局不是某一种技术"赢了"——而是围绕模型的认知特点，建立起一套**从原始知识到模型有效消费**的完整转译体系。在这个体系中：

- **Skill** 是知识的"操作手册"——教模型怎么做
- **Memory** 是知识的"经验日志"——记住模型该避免什么、注意什么
- **Wiki** 是知识的"百科全书"——让模型理解领域背景
- **工具** 是知识的"实时传感器"——让模型感知当前状态

四者组合，模型就不再是一个"什么都不知道的通用推理机"，而是一个"深度理解你的业务、熟悉你的流程、记住你的偏好、能随时查看实时状态"的领域专家。

这，才是"推理时计算"这个最好的厨师所需要的"食材基础设施"。当厨师足够强大、食材足够充分时，智能体的能力就不再受限于任何一个单一因素，而是开始展现真正的复合智能。

> 推理时计算让模型"想得更深"，知识基础设施让模型"知道更多"。前者是能力，后者是根基。能力会被追赶，根基会持续加固。对企业而言，建设自己的模型原生知识体系——而非追逐最新的模型或最精巧的 RAG 架构——才是 AI 时代最值得投入的长期工程。
